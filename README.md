# BehaviorDrift

BehaviorDrift watches a GitHub repository's *weekly activity pattern* : commit volume, 
number of unique authors, dependency-file churn , and flags weeks that deviate from that
specific repo's own history. It's not a vulnerability scanner and doesn't match known
signatures (that's what Dependabot/Snyk already do); it's looking for the kind of thing a
signature can't catch: a repo that's normally three people doing ten commits a week
suddenly having one person do forty commits and touch six dependency files.

 Link:https://behaviordrift.onrender.com/

## How it works

```
collector.py   → pulls commits from the GitHub API, buckets them into calendar weeks
vectorizer.py  → turns each week into a numeric vector (commit_count, unique_authors,
                  dependency_touches, commits_per_author)
detector_ml.py → primary detector: an IsolationForest trained on a repo's own prior
                  weeks (rolling baseline — a week is only ever scored against weeks
                  strictly before it, never its own future)
detector.py    → secondary, fully transparent z-score layer; explains *which* features
                  drove a flag and by how much, even though the ML model makes the
                  primary flag/no-flag decision
agent.py       → for flagged weeks only, asks an LLM (Claude, falling back to Gemini,
                  falling back to a deterministic template) to turn the statistical
                  evidence + real commit messages into a plain-English explanation
memory.py      → stores each week's fingerprint (Qdrant if configured, otherwise a
                  local JSON file) so future weeks have history to compare against
pipeline.py    → wires all of the above together end to end
report.py      → renders a pipeline result as a self-contained HTML report
```

## Repo layout

```
src/        core pipeline (see above) — no framework dependencies, pure Python
backend/    FastAPI wrapper around src/pipeline.py (main.py)
frontend/   React + Vite UI that talks to the backend
demo/       CLI entry point (run_demo.py) — works fully offline
tests/      pytest suite covering src/ and report.py
data/       local_memory.json — the local (non-Qdrant) fingerprint store
```

## Quickstart: CLI, no setup required

```bash
pip install -r requirements.txt
python demo/run_demo.py --offline
```

This runs the full pipeline against synthetic sample data (no GitHub API calls, no LLM
keys needed) and writes `report.html`. Open it in a browser to see the result.

To run against a real repo instead:

```bash
python demo/run_demo.py --repo pallets/flask
```

## Running the full stack (backend + frontend)

**Backend:**

```bash
cd backend
pip install -r requirements.txt
uvicorn main:app --reload --port 8000
```

**Frontend** (separate terminal):

```bash
cd frontend
npm install
npm run dev
```

Then open the Vite dev server URL (typically `http://localhost:5173`) — its dev proxy
forwards `/api/*` requests to the backend on port 8000, so no extra config is needed
locally.

## Environment variables

None of these are required to run the offline demo. Set only the ones relevant to what
you're doing, as a `.env` file in the project root (loaded automatically by
`backend/main.py`) or as regular environment variables for the CLI.

| Variable | Required for | Notes |
|---|---|---|
| `GITHUB_TOKEN` | Scanning real repos | Without it you're limited to ~60 unauthenticated GitHub API requests/hour. A classic PAT with no special scopes is enough for public repos. |
| `ANTHROPIC_API_KEY` | LLM-written explanations via Claude | First choice in the fallback chain. Without it (and without `GEMINI_API_KEY`) explanations are generated by a deterministic template instead — no external calls made. |
| `GEMINI_API_KEY` | LLM-written explanations via Gemini | Used only if `ANTHROPIC_API_KEY` is unset or the Claude call fails. |
| `QDRANT_URL` | Persistent shared memory store | Without it, fingerprints are stored in `data/local_memory.json` instead. |
| `QDRANT_API_KEY` | Qdrant, if your instance requires auth | Ignored if `QDRANT_URL` isn't set. |
| `CORS_ORIGINS` | Backend, if serving a frontend from a non-default origin | Comma-separated list; defaults to `http://localhost:5173`. |

## Running the tests

```bash
pip install -r requirements.txt
python -m pytest tests/ -v
```

The suite is fully offline: GitHub API calls are mocked (`requests-mock`), LLM calls are
mocked, and each test that touches the memory store uses an isolated scratch file rather
than `data/local_memory.json`.
